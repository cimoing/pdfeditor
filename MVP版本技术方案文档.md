# PDF 编辑器 MVP 版本技术方案文档

## 1. 方案目标

MVP 版本目标是在 Windows、macOS、Linux 桌面端提供基础可用的 PDF 查看与编辑能力，重点支持 PDF 文本和图片编辑，同时保证应用在老旧设备上具备可接受的启动速度、内存占用和操作流畅度。

本阶段不使用 Tauri、Electron、WebView 等基于浏览器运行时的桌面技术，优先采用原生桌面技术和高性能 PDF 处理库。

## 2. 技术原则

- 使用原生桌面技术，不依赖浏览器内核作为应用运行时。
- 控制内存占用，避免一次性加载整份 PDF 的全部页面和资源。
- 控制 CPU 占用，采用按需渲染、缓存复用和后台任务调度。
- 优先保证 PDF 查看、编辑、保存链路稳定，再扩展高级能力。
- 核心 PDF 处理能力应平台无关，UI 层按平台适配。
- 为后续 OCR、页面管理、移动端扩展预留模块边界。

## 3. 技术选型

### 3.1 应用开发语言

推荐使用 C++ 作为核心开发语言。

原因：

- 性能稳定，适合 PDF 渲染、编辑和文件处理。
- 内存控制能力强，便于针对老旧设备做优化。
- 跨平台生态成熟，可同时支持 Windows、macOS、Linux。
- 容易接入成熟 PDF 引擎和系统原生能力。

### 3.2 桌面 UI 框架

推荐方案：Qt Widgets。

原因：

- 原生桌面应用体验成熟。
- 跨平台支持 Windows、macOS、Linux。
- 相比基于浏览器的桌面方案，运行时资源占用更低。
- 对老旧设备更友好，启动速度和内存占用更可控。
- 适合构建工具栏、缩略图栏、页面视图、属性面板等传统办公软件界面。

不推荐方案：

- 不使用 Tauri。
- 不使用 Electron。
- 不使用基于 WebView 的桌面应用壳。
- 不以浏览器渲染层作为主要 UI 技术。

### 3.3 PDF 渲染与编辑引擎

候选方案：

- MuPDF
- PDFium
- PoDoFo

MVP 推荐组合：

- PDF 渲染：MuPDF 或 PDFium。
- PDF 结构解析与保存：优先评估 MuPDF 的编辑能力；如编辑能力不足，可引入专门的 PDF 修改库。

选型关注点：

- 文本对象读取、定位与修改能力。
- 图片对象读取、替换、插入与删除能力。
- 页面渲染速度。
- 内存占用。
- 跨平台编译难度。
- 开源协议和商业使用限制。

MVP 阶段建议先完成技术验证，再最终确认 PDF 引擎。

## 4. 总体架构

应用采用分层架构：

- UI 层：负责窗口、菜单、工具栏、页面视图、缩略图、属性面板和用户交互。
- 文档服务层：负责 PDF 文件打开、页面管理、对象查询、编辑命令、保存。
- 渲染层：负责页面位图渲染、缩放渲染、局部刷新、缓存管理。
- 编辑模型层：负责文本对象、图片对象、选区、变换、撤销重做。
- 文件 I/O 层：负责本地文件读取、临时文件、保存、另存为、异常恢复。
- 平台适配层：负责不同系统的文件选择器、字体、快捷键、路径、权限差异。

## 5. MVP 功能实现方案

### 5.1 PDF 打开与浏览

实现方式：

- 打开 PDF 时只解析文档目录和必要元数据。
- 页面内容按需加载。
- 当前页和相邻页面优先渲染。
- 缩略图延迟生成，滚动到可见区域时再渲染。

目标：

- 降低大文件打开时的等待时间。
- 避免一次性占用大量内存。
- 在低配置设备上保持可用。

### 5.2 页面渲染

实现方式：

- 页面渲染使用后台线程，避免阻塞 UI。
- 根据当前缩放比例渲染页面位图。
- 缓存最近访问页面的渲染结果。
- 当内存压力较大时主动释放远离当前视口的页面缓存。
- 快速滚动时优先显示低分辨率预览，再补充高清渲染。

缓存策略：

- 默认缓存当前页、前后各 1 到 2 页。
- 缩略图缓存设置独立上限。
- 大尺寸页面或高缩放比例下限制最大渲染位图尺寸。

### 5.3 文本编辑

实现方式：

- 解析页面中的文本对象及其边界框。
- 用户点击页面时，根据坐标命中文本对象。
- 使用覆盖式编辑控件进行文本修改。
- 修改完成后更新 PDF 文本对象或写入新的文本对象。
- 对无法安全修改的复杂文本对象，提示用户改用新增文本框覆盖。

MVP 支持范围：

- 修改简单文本对象内容。
- 新增文本框。
- 移动文本对象。
- 删除文本对象。
- 修改基础样式，如字号、颜色。

限制：

- 不承诺完整还原所有复杂字体和特殊编码文本。
- 不支持扫描件图片中文字直接编辑。
- 不支持复杂段落自动重排。

### 5.4 图片编辑

实现方式：

- 解析页面图片对象及其边界框。
- 支持图片对象选择、移动、缩放、删除。
- 插入图片时将图片编码为 PDF 支持的图像对象。
- 替换图片时保留原图片边界框和变换矩阵。

MVP 支持范围：

- 插入 PNG、JPG、JPEG。
- 替换现有图片。
- 移动图片位置。
- 缩放图片尺寸。
- 删除图片对象。

### 5.5 保存与另存为

实现方式：

- 编辑操作先记录到内存中的编辑命令模型。
- 保存时生成新的 PDF 文件，避免直接破坏原文件。
- 覆盖原文件前先写入临时文件，成功后再替换。
- 保存失败时保留当前编辑状态，并提示错误原因。

目标：

- 降低文件损坏风险。
- 支持后续扩展撤销、重做和自动恢复。

### 5.6 撤销与重做

实现方式：

- 使用命令模式记录编辑操作。
- 每个编辑命令包含执行和回滚逻辑。
- 文本修改、对象移动、图片插入、图片删除都作为独立命令。

资源控制：

- 撤销栈设置最大步数。
- 大图片操作不在撤销栈中重复保存完整图片数据，优先保存引用或临时文件路径。

## 6. 老旧设备优化策略

### 6.1 内存优化

- PDF 页面按需加载，不全量加载。
- 页面渲染结果设置缓存上限。
- 缩略图使用低分辨率图像。
- 大图片插入前进行尺寸检查和必要压缩。
- 撤销栈限制最大数量。
- 大文件保存使用流式处理，避免完整文件多份拷贝常驻内存。

建议目标：

- 空文档启动内存尽量控制在 150 MB 以内。
- 打开常规 PDF 后内存尽量控制在 300 MB 以内。
- 大文件场景允许更高占用，但应有缓存释放机制。

### 6.2 CPU 优化

- 页面渲染放到后台线程。
- 快速滚动时合并渲染请求，取消不可见页面渲染任务。
- 缩放过程中使用已有缓存预览，用户停止操作后再高清重绘。
- 缩略图分批生成，避免打开文件后集中占用 CPU。
- 避免频繁全页面重绘，优先做局部刷新。

### 6.3 启动优化

- 应用启动时只加载必要模块。
- PDF 引擎按需初始化。
- OCR、高级编辑等后续模块不进入 MVP 启动链路。
- 最近文件缩略图延迟加载。

### 6.4 磁盘与临时文件优化

- 大图片和大对象可写入临时文件，避免长期占用内存。
- 临时文件集中管理，应用退出时清理。
- 异常退出后下次启动检查并清理过期临时文件。

## 7. 多线程模型

建议线程模型：

- UI 主线程：处理交互和界面刷新。
- 渲染线程池：处理页面渲染和缩略图渲染。
- 文件任务线程：处理打开、保存、导出等文件 I/O。

约束：

- UI 线程不执行耗时 PDF 解析和渲染。
- 同一 PDF 文档的写操作需要串行化，避免状态冲突。
- 渲染任务应支持取消，避免无效任务堆积。

## 8. 目录结构建议

建议项目结构：

```text
pdfeditor/
  docs/
    PDF编辑器桌面应用需求文档.md
    MVP版本技术方案文档.md
  src/
    app/
    ui/
    document/
    render/
    edit/
    platform/
    common/
  tests/
    unit/
    integration/
  third_party/
  tools/
```

模块说明：

- `app`：应用入口、生命周期、配置管理。
- `ui`：窗口、工具栏、页面视图、属性面板。
- `document`：PDF 文档加载、保存、页面管理。
- `render`：页面渲染、缩略图、缓存。
- `edit`：文本编辑、图片编辑、撤销重做。
- `platform`：系统差异适配。
- `common`：通用数据结构、日志、错误处理。

## 9. 技术验证计划

MVP 开发前建议完成以下验证：

- PDF 引擎能否稳定打开常见 PDF。
- 是否可以获取文本对象坐标和内容。
- 是否可以修改或新增文本对象并保存。
- 是否可以获取、替换、插入图片对象。
- 渲染 100 页以上 PDF 的内存和 CPU 表现。
- 在低配置设备或虚拟机中验证启动速度和翻页流畅度。
- 编辑后的 PDF 是否能被主流阅读器打开。

## 10. MVP 里程碑

- 完成 PDF 引擎对比。
- 完成 Qt Widgets 原型。
- 完成 PDF 打开、单页渲染和保存验证。

### 10.2 基础阅读器

- 支持打开 PDF。
- 支持页面浏览、缩放、跳转。
- 支持缩略图导航。
- 支持基础搜索。

### 10.3 文本编辑

- 支持文本对象选择。
- 支持新增文本框。
- 支持修改简单文本对象。
- 支持文本移动、删除和基础样式调整。

### 10.4 图片编辑

- 支持图片对象选择。
- 支持插入、替换、移动、缩放和删除图片。

### 10.5 保存与稳定性

- 支持保存和另存为。
- 支持撤销和重做。
- 支持异常提示。
- 完成基础兼容性测试和老旧设备性能测试。

## 11. 风险与应对

### 11.1 PDF 编辑复杂度高

风险：PDF 不是天然面向编辑的格式，不同文件结构差异较大。

应对：

- MVP 明确支持范围，优先支持简单文本和图片对象编辑。
- 对复杂对象提供提示或覆盖式编辑方案。
- 后续逐步增强复杂 PDF 兼容性。

### 11.2 字体和编码兼容问题

风险：部分 PDF 使用嵌入字体、子集字体或特殊编码，直接修改文本可能导致乱码或排版异常。

应对：

- 优先支持可识别、可替换的简单文本对象。
- 对复杂字体场景使用新增文本覆盖方案。
- 保存前提供预览和兼容性提示。

### 11.3 老旧设备性能不足

风险：大文件、高分辨率页面和复杂图片可能导致内存或 CPU 占用过高。

应对：

- 按需加载和渲染。
- 限制缓存大小。
- 渲染任务可取消。
- 大图插入前压缩。
- 提供低资源模式。

### 11.4 跨平台差异

风险：不同系统在字体、文件权限、窗口行为、输入法和快捷键上存在差异。

应对：

- 核心逻辑保持平台无关。
- 平台相关能力集中在适配层。
- 每个平台建立基础回归测试清单。

## 12. MVP 不包含范围

以下能力不纳入 MVP：

- OCR。
- 扫描件中文字直接编辑。
- 表单字段编辑。
- 电子签名。
- 云端同步。
- 多人协作。
- 批量处理。
- 高级页面重排。
- 复杂段落自动排版。

## 13. Qt 方案对商业化的影响

Qt 可以用于商业化产品，但商业化影响主要取决于所选择的 Qt 许可证、使用的 Qt 模块、链接方式和产品分发方式。

### 13.1 可选授权路径

Qt 通常有两类使用路径：

- 使用 Qt 商业许可证。
- 使用 Qt 开源许可证，如 LGPL 或 GPL。

如果产品计划闭源销售，并且希望最大程度降低开源合规风险，建议优先评估 Qt 商业许可证。

如果使用 Qt LGPL 版本，通常可以开发闭源应用，但必须满足 LGPL 对用户权利、库替换、许可证声明、源码或对象文件提供等方面的要求。具体合规要求需要由法务或开源合规人员确认。

如果使用 Qt GPL 模块，则可能要求整个应用以 GPL 兼容方式开源发布，不适合闭源商业软件。

### 13.2 对闭源商业软件的影响

使用 Qt 开发闭源商业 PDF 编辑器时，需要重点关注：

- 避免使用仅 GPL 授权的 Qt 模块。
- 优先使用 LGPL 可用模块或购买商业许可证。
- 避免静态链接 LGPL Qt 库，除非能够满足重新链接等合规要求。
- 保留 Qt 许可证声明、版权声明和第三方组件声明。
- 如果修改 Qt 源码，需要按许可证要求处理修改内容。
- 发布安装包时，需要检查是否包含 Qt 第三方依赖及其许可证义务。

### 13.3 对成本的影响

商业许可证会增加直接授权成本，但可以降低以下隐性成本：

- 开源许可证合规成本。
- 法务审查成本。
- 模块可用性限制。
- 后续版本升级和长期维护风险。
- 企业客户采购时的合规疑虑。

对于面向企业客户、政府客户或收费闭源软件的产品，商业许可证通常更利于销售、审计和长期维护。

### 13.4 对技术选型的影响

如果继续选择 Qt Widgets，建议 MVP 阶段遵循以下策略：

- 只使用 Qt Core、Qt GUI、Qt Widgets 等基础模块。
- 避免使用 Qt WebEngine、Qt Multimedia 等可能引入复杂第三方依赖的模块。
- 不使用仅 GPL 授权的 Qt 模块。
- 默认采用动态链接方式。
- 在技术预研阶段建立 Qt 模块清单和许可证清单。
- 在正式商业发布前完成开源合规审查。

### 13.5 建议决策

建议将 Qt 作为 MVP 的候选技术方案之一，但在立项阶段同步完成授权评估。

推荐结论：

- 如果产品确定闭源商业化，优先考虑购买 Qt 商业许可证。
- 如果早期预算有限，可以先基于 LGPL 模块完成技术原型，但需要严格限制模块范围和链接方式。
- 如果后续商业化面向企业客户，建议尽早切换或确认商业许可证，避免产品发布前产生授权风险。

### 13.6 替代方案预案

如 Qt 授权成本或合规要求不可接受，可评估以下替代方案：

- wxWidgets：原生跨平台 UI，许可证对商业闭源更友好，但 UI 开发体验和生态需评估。
- FLTK：轻量、资源占用低，但界面能力相对基础。
- 平台原生 UI：Windows、macOS、Linux 分别实现，商业授权风险低，但开发和维护成本高。
- Dear ImGui：适合工具型界面和内部工具，但不适合传统办公软件体验。

从 MVP 研发效率、跨平台能力、老旧设备可用性和商业化风险综合看，Qt 仍然是强候选，但需要在许可证层面提前做决策。

## 14. Rust 方案可行性分析

Rust 可以作为 PDF 编辑器 MVP 的核心技术候选，尤其适合实现 PDF 文档处理、渲染调度、缓存管理、文件 I/O 和跨平台业务逻辑。但如果要求完整替代 C++/Qt 构建成熟桌面办公软件界面，需要谨慎评估 Rust GUI 生态成熟度和 PDF 编辑库能力。

### 14.1 总体结论

Rust 方案可行，但更适合作为“核心层优先”的技术路线：

- 用 Rust 实现 PDF 文档服务、编辑模型、渲染任务调度、缓存、撤销重做、文件保存等核心逻辑。
- PDF 渲染和底层编辑能力可通过绑定 PDFium、MuPDF 等成熟 C/C++ PDF 引擎实现。
- UI 层可选择 Rust 原生 GUI 框架，或在商业化和成熟度要求较高时使用 Qt/C++ 作为 UI 层。

不建议在 MVP 阶段完全依赖纯 Rust PDF 库实现复杂 PDF 编辑能力。PDF 格式复杂，文本、字体、图片对象和内容流编辑都需要成熟引擎支撑。

### 14.2 Rust 的优势

- 内存安全性高，适合处理复杂文件格式，能降低崩溃和内存错误风险。
- 性能接近 C/C++，适合 PDF 渲染调度、缓存管理和大文件处理。
- 并发模型较好，适合实现后台渲染、任务取消、异步文件 I/O。
- 工具链现代化，依赖管理、测试、格式化和静态检查体验较好。
- 跨平台能力强，可以同时支持 Windows、macOS、Linux。
- 对老旧设备友好，不需要浏览器运行时，内存和启动成本可控。

### 14.3 Rust 的主要风险

- 纯 Rust PDF 编辑生态不如 C/C++ 成熟。
- PDF 中文本对象修改、字体处理、内容流重写、图片替换等能力实现复杂。
- Rust 桌面 GUI 生态整体仍弱于 Qt Widgets、原生平台 UI 等成熟方案。
- 如果使用 PDFium、MuPDF 等 C/C++ 引擎，仍需要处理 FFI、动态库分发和跨平台构建问题。
- 商业化时仍需审查 PDF 引擎、GUI 框架和第三方 crate 的许可证。

### 14.4 PDF 处理技术路线

可选路线：

- `pdfium-render`：Rust 对 PDFium 的高级封装，可用于 PDF 渲染、文本和图片提取、部分编辑能力验证。
- `lopdf`：纯 Rust PDF 操作库，适合低层 PDF 对象处理、合并、拆分、结构修改，但不适合单独承担复杂可视化编辑。
- MuPDF C API：性能和能力较强，但 Rust 中通常需要通过 FFI 使用。

MVP 推荐路线：

- 优先验证 `pdfium-render + PDFium` 是否满足页面渲染、文本定位、图片提取和基础编辑需求。
- 如需要更底层的 PDF 对象修改，可结合 `lopdf` 做补充。
- 不建议 MVP 自研 PDF 内容流解析和重写能力。

### 14.5 UI 技术路线

Rust UI 候选：

- Slint：适合跨平台原生 UI，支持 Rust，商业授权路径相对清晰，适合轻量桌面应用。
- egui：开发效率高、资源占用低，适合工具型应用，但传统办公软件质感和复杂控件能力需要评估。
- iced：纯 Rust GUI 框架，适合跨平台应用，但复杂桌面办公软件成熟度需要验证。

如果产品对传统桌面办公体验要求较高，仍建议保留以下混合方案：

- UI 层使用 Qt Widgets。
- 核心文档处理层使用 Rust。
- 通过 C ABI 或桥接层连接 Qt/C++ 与 Rust 核心库。

该方案可以兼顾 Qt UI 成熟度和 Rust 核心安全性，但会增加构建和调试复杂度。

### 14.6 对老旧设备的影响

Rust 方案本身适合老旧设备：

- 不依赖浏览器运行时，启动成本低。
- 可精确控制内存分配和缓存生命周期。
- 可实现按需加载、后台渲染、任务取消和低资源模式。
- 编译为原生二进制，运行时依赖少。

需要注意：

- GUI 框架选择会直接影响内存和 CPU 占用。
- PDFium 或 MuPDF 的动态库体积和运行时资源占用需要实测。
- 图片插入、页面高倍率渲染、大文件保存仍可能产生资源峰值，需要通过缓存上限和流式处理控制。

### 14.7 商业化影响

Rust 语言本身不会增加商业授权成本。

但商业化仍需关注：

- PDF 引擎许可证。
- GUI 框架许可证。
- 第三方 crate 许可证。
- 动态库分发授权。
- 是否修改了开源库源码。

一般来说，Rust crate 常见 MIT、Apache-2.0 等宽松许可证较多，对闭源商业软件友好。但具体到 PDF 引擎和 GUI 框架，必须逐项确认。

### 14.8 推荐决策

如果团队具备 Rust 能力，建议将 Rust 纳入 MVP 技术预研，并采用以下优先级：

1. 首选：Rust 核心层 + PDFium/MuPDF + Qt 或 Slint UI。
2. 次选：Rust 全栈 + Slint UI + PDFium。
3. 谨慎选择：Rust 全栈 + egui，用于工具型产品可以，但需验证办公软件体验。
4. 不建议：完全基于纯 Rust PDF 库自研复杂 PDF 编辑能力。

最终建议：

- 若目标是尽快做出成熟桌面商业软件，Qt/C++ 方案更稳。
- 若目标是长期构建安全、轻量、可维护的跨平台核心能力，Rust 很有价值。
- MVP 最理性的路线是先做 Rust + PDFium 技术原型，验证文本和图片编辑能力，再决定 UI 层使用 Qt、Slint 或其他方案。
